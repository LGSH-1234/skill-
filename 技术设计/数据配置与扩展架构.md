# 数据配置与扩展架构

## 一、推荐技术栈
- 前端框架：Next.js
- 样式方案：Tailwind CSS
- 内容管理：本地内容文件驱动
- 部署建议：Vercel

## 二、为什么采用内容文件驱动
当前需求核心是稳定展示和后续高频新增 Skill，且维护方式是由 Claude 直接协助改代码上传，因此内容文件驱动是最合适的方案：
- 简单直接
- 不需要先做后台
- 结构清晰
- 后续可平滑升级到 CMS

## 三、目录建议

```bash
/src
  /app
  /components
  /lib
  /types
/content
  /skills
    /zh
    /en
/public
  /downloads
/产品规划
/技术设计
/讨论记录
/项目概览
```

## 四、Skill 内容模型建议
建议每个 Skill 对应一个独立内容文件，推荐字段：

```ts
export interface SkillItem {
  slug: string
  title: string
  summary: string
  scenarios: string[]
  howToUse: string[]
  download?: {
    label: string
    href: string
  }
  category?: string
  tags?: string[]
  featured?: boolean
  publishedAt?: string
  locale: 'zh' | 'en'
}
```

## 五、内容组织建议
### 中文优先
首版以 `/content/skills/zh` 为主。

### 双语预留
未来增加 `/content/skills/en`，通过 locale 与路由控制多语言展示。

## 六、页面层结构建议
- `/`：首页
- `/skills`：Skill 列表页
- `/skills/[slug]`：Skill 详情页
- `/about`：关于页

如未来做双语，可扩展为：
- `/zh`
- `/zh/skills/[slug]`
- `/en`
- `/en/skills/[slug]`

## 七、模板下载建议
- 下载文件统一放在 `/public/downloads`
- 内容文件中仅维护下载链接与按钮文案
- 后续可加入下载埋点

## 八、扩展路线
### 从内容文件驱动升级到 CMS 的时机
当出现以下情况时，可考虑接入 Headless CMS：
- Skill 数量快速增长
- 需要多人协作编辑
- 需要可视化内容管理
- 需要发布审核流程

### 可选升级方向
- Notion/CMS 同步
- 下载统计
- SEO 自动化
- 标签与搜索系统
