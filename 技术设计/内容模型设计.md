# 内容模型设计

## 一、目标
将 `/Users/lgsh/Desktop/skills` 中已经沉淀好的 Skill 目录，映射为网站可直接消费的结构化内容源，用于驱动：
- 首页精选 Skill
- Skill 列表页
- Skill 详情页
- 后续双语扩展

首版不直接把整个目录原样暴露到前端，而是从现有 Skill 资产中抽取适合展示的公开字段，形成网站专用内容模型。

## 二、现有 Skill 资产结构侦查结论
桌面 `skills` 目录中的每个 Skill 基本采用如下结构：

```bash
/skill-name
  SKILL.md
  /references
  /scripts
  LICENSE.txt
  install.sh
```

其中：
- `SKILL.md` 是最核心的内容源
- `references/` 主要是补充资料、模板、规范说明
- `scripts/`、`install.sh` 主要是实现细节，不适合作为首版网站正文直接展示
- 部分 Skill 带有 LICENSE、CLAUDE.md、AGENTS.md 等辅助文件，首版可忽略

## 三、网站内容模型建议
建议在网站项目中建立一层“展示模型”，不要直接把原始 `SKILL.md` 结构硬绑到页面。

```ts
export interface SkillContent {
  slug: string
  locale: 'zh'
  title: string
  summary: string
  category: string
  tags: string[]
  featured: boolean
  heroNote?: string
  scenarios: string[]
  howToUse: string[]
  triggerPhrases?: string[]
  outputs?: string[]
  body: string
  references?: Array<{
    title: string
    path: string
  }>
  download?: {
    label: string
    href: string
  }
}
```

## 四、字段映射关系

### 1. 直接可从 `SKILL.md` 提取的字段
- `slug`：目录名，如 `ai-companion-prd`
- `title`：优先取正文一级标题，如“AI 情感陪伴产品 PRD 生成器”
- `summary`：优先取 frontmatter `description`
- `body`：`SKILL.md` 主体内容
- `triggerPhrases`：从“触发词 / 触发场景 / 触发方式”提取
- `howToUse`：从“使用流程 / 工作流程 / 核心工作流”提取关键步骤
- `outputs`：从“输出 / 输出要求 / 输出文档”提取

### 2. 需要人为补齐或二次加工的字段
- `category`：用于列表页分组与筛选
- `tags`：用于增强浏览效率
- `featured`：用于首页精选
- `heroNote`：用于详情页顶部的品牌化短句
- `download`：如果该 Skill 有适合公开下载的模板，再配置下载按钮

## 五、分类体系建议
结合现有 13 个 Skill，首版建议采用以下分类：
- 产品文档生成
- 流程图与工作流
- Prompt 与 AI 交互
- 开发与工程实践
- 测试与效率工具

示例：
- `ai-companion-prd` → 产品文档生成
- `feature-iteration-prd` → 产品文档生成
- `dify-workflow-builder` → 流程图与工作流
- `prompt-optimizer` → Prompt 与 AI 交互
- `webapp-testing` → 测试与效率工具
- `next-best-practices` → 开发与工程实践

## 六、页面映射建议

### 1. 首页
展示内容：
- 品牌主视觉
- 个人能力概览
- 精选 Skill（featured=true）
- Skill 分类入口
- 关于我入口

### 2. Skill 列表页
展示字段：
- 标题
- 简介
- 分类
- 标签
- 是否精选

支持后续扩展：
- 分类筛选
- 标签筛选
- 搜索

### 3. Skill 详情页
推荐模块顺序：
1. Hero 区：标题 + summary + category/tag
2. 适用场景：`scenarios`
3. 使用方法：`howToUse`
4. 输出结果：`outputs`
5. 详细说明：`body`
6. 延伸资料：`references`
7. 下载按钮：`download`

## 七、关于 references / scripts 的展示策略

### references/
建议在详情页中作为“延伸资料”展示，而不是主内容。
展示方式：
- 标题列表
- 可点进查看原文或下载
- 不抢正文层级

### scripts/、install.sh
首版默认不展示。
原因：
- 面向普通访客可读性弱
- 会削弱品牌展示感
- 更适合作为实现层资产保留在仓库中

如未来希望增强“技术可信度”，可以在详情页底部加“实现补充”区域，再选择性开放。

## 八、内容接入策略建议
建议在网站项目中新增一层规范化内容目录：

```bash
/content
  /skills
    /zh
      ai-companion-prd.md
      feature-iteration-prd.md
```

即：
1. 从桌面 `skills/*/SKILL.md` 提取核心内容
2. 转换成网站统一字段格式
3. 后续网站只读取 `/content/skills/zh`，而不是直接耦合桌面原目录

这样做的好处：
- 页面结构更稳定
- 可对原始 Skill 内容做品牌化改写
- 便于后续补充 featured、category、tags、download 等展示字段
- 便于未来新增英文内容 `en`

## 九、首版落地建议
首版先完成：
1. 建立统一内容 schema
2. 选 6~8 个代表性 Skill 先接入
3. 首页只展示 3~4 个精选 Skill
4. 详情页优先保证可读性和品牌感，不追求把所有原始说明都完全平铺
